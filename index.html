ABOUT

자기소개

저는 플레이어가 실제로 체감하는 규칙과 흐름을 우선으로 보는 기획자를 지향합니다. 보기 좋은 설정만 정리하는 것이 아니라, 플레이 중 어떤 혼란이 발생하는지, 어떤 정보가 늦게 전달되는지, 어떤 수치 곡선에서 이탈이 생기는지를 구조적으로 분해해 문서화하는 쪽에 강점이 있습니다.

포트폴리오에서는 결과물 자체보다도 문제 정의 방식, 해결 논리, 문서 전달력이 드러나도록 구성했습니다. 따라서 각 프로젝트는 이미지 중심 쇼케이스가 아니라, 실무 검토에 가까운 문서형 구조로 정리했습니다.

CORE STRENGTHS

핵심 역량

시스템 분석

규칙 간 충돌, 피드백 누락, 성장 체감 저하 같은 문제를 구조 단위로 분리해 파악합니다.

문서 구조화

기획 의도, 기능 정의, 예외 처리, QA 포인트를 타 부서가 빠르게 읽을 수 있는 형식으로 정리합니다.

문제 해결 서술

무엇이 문제였는지, 왜 그렇게 판단했는지, 수정 후 어떤 변화가 있었는지까지 연결해서 설명합니다.

협업 가독성

한 번에 이해되지 않는 문서는 실무 전달력이 낮다고 판단하고, 표와 계층 구조를 적극 사용합니다.

PROJECT 01

LoLLing Dice

주사위 기반 전투 시스템 기획 프로젝트

유형
팀 프로젝트 / 전투 시스템 기획
기간
2025.XX ~ 2025.XX
역할
시스템 기획, 밸런싱, 문서화
사용 툴
Unity, Google Sheets, Figma

프로젝트 개요

주사위 결과를 전투 규칙에 직접 연결하는 구조를 중심으로 설계한 프로젝트입니다. 단순한 랜덤성보다는 결과 해석과 선택 구조에 초점을 맞췄고, 플레이어가 주사위 값을 읽고 다음 행동을 예측할 수 있도록 규칙을 정리했습니다.

내가 맡은 일

  • 전투 규칙 문서 초안 작성 및 예외 처리 정리
  • 레벨/경험치 곡선 검토 및 중반 구간 성장 체감 조정
  • 테스트 관점에서 혼란이 큰 규칙을 분리해 재작성

문제 → 해결 → 결과

LoLLing Dice 문제 해결 표
문제 해결 결과
주사위 결과와 행동 우선순위 관계가 직관적으로 보이지 않았다. 행동 순서를 상태도와 표로 다시 정리하고, 예외 상황을 문서에서 분리했다. 규칙 이해에 필요한 왕복 질문이 줄고, 테스트 피드백 수집이 쉬워졌다.
중반 성장 체감이 약해 전투 보상이 밋밋하게 느껴졌다. 경험치 요구량과 보상 곡선을 구간별로 다시 잡아 체감 상승 폭을 조정했다. 레벨업 간격이 덜 답답해지고, 성장 구간의 이탈 가능성을 줄일 수 있었다.

PROJECT 02

Boss Pattern Review

보스 패턴 전달력과 학습 난이도 분석

유형
분석형 포트폴리오
기간
2025.XX
역할
패턴 분석, 피드백 구조 정리
사용 툴
Figma, 문서화 도구, 영상 캡처

프로젝트 개요

보스전에서 플레이어가 정보를 어떻게 읽고, 어디서 오판하는지를 중심으로 패턴 전달 구조를 분석한 작업입니다. 난이도 자체보다도 “왜 납득되지 않는지”를 설명하는 데 집중했습니다.

내가 맡은 일

  • 패턴 진입 예고, 위험 반경, 타격 타이밍 정보를 분리해 분석
  • 시각/행동 피드백이 겹치는 구간을 체크해 읽기 순서를 재설계
  • 플레이어 학습 관점에서 패턴 난이도를 단계별로 재정리

문제 → 해결 → 결과

Boss Pattern Review 문제 해결 표
문제 해결 결과
위험 신호가 공격 판정 직전에만 보여 학습이 어려웠다. 예고 타이밍, 시각 강조, 행동 준비 동작을 별도 층으로 구분해 정리했다. 플레이어가 패턴 해석에 필요한 단서를 더 일찍 확보하는 구조를 제안할 수 있었다.
같은 화면에 너무 많은 정보가 겹쳐 핵심 위험이 묻혔다. 우선순위가 낮은 연출 정보와 치명적 위험 신호를 분리했다. 정보 밀도를 낮추면서도 보스전 긴장감은 유지하는 방향을 도출했다.

검토 포인트

  • 보스전은 난이도보다 납득 가능성이 먼저 확보되어야 한다고 판단했습니다.
  • 연출과 판정 전달이 충돌할 때는 연출보다 판정 가독성을 우선했습니다.

PROJECT 03

QA Process Template

기획 문서와 테스트 케이스 연결 구조 정리

유형
문서/프로세스 설계
기간
2025.XX
역할
QA 시트 구조 설계, 항목 정리
사용 툴
Google Sheets, 문서 도구

프로젝트 개요

테스트 케이스와 버그 리포트 구조가 과하게 중복되면 관리 비용이 늘어난다고 판단해, 최소한의 열 구조로도 추적이 가능하도록 QA 문서 체계를 정리한 작업입니다.

내가 맡은 일

  • 테스트 항목, 실행 결과, 버그 기록의 역할 범위를 재정의
  • 반복 입력이 많은 열을 줄이고 필수 기록만 남기는 방향으로 정리
  • 실무에서 빠르게 확인 가능한 시트 허브 구조 제안

문제 → 해결 → 결과

QA Process Template 문제 해결 표
문제 해결 결과
테스트 케이스와 버그 리포트 열 구성이 지나치게 중복되었다. 목적이 다른 기록은 분리하고, 공통 항목만 최소 범위로 유지했다. 시트 가로 폭을 줄이면서도 필요한 추적 정보는 유지할 수 있었다.
QA 실행자가 어디에 무엇을 적어야 하는지 한눈에 이해하기 어려웠다. 시트별 역할을 허브 구조로 재정리하고 입력 규칙을 명시했다. 문서 사용 진입 장벽을 낮추고, 기록 누락 가능성을 줄이는 방향을 확보했다.
